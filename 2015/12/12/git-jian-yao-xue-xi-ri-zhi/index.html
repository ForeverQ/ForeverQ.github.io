git 简要学习日志

发表于 2015-12-12   |   分类于 学习

前两天在 Codecademy 在线学习了 git 课程，感觉颇为实用，也消除了我对 git 的一些困惑。

git 基本操作

初始化

init 意思是初始化，作用是设置所有 git 跟踪文件改动所需的组件，表现上来看，会在文件夹内生成一个 .git 隐藏文件。

概念模型

git 的基本逻辑分为三个部分：

  • Working Directory，文件系统的概念，创建、编辑、删除和组织文件的空间。
  • Staging Area，展现所有改动的空间，实际的 git 操作都是以此为中心。
  • Repository，对应 Github 中 Repo 的概念，储存整个项目所有改动版本。

git 的 workflow 主要包括：

  1. 在 working directory 中改动文件；
  2. 添加这些文件到 staging area 中；
  3. 通过 commit 存入 repo。

git 版本回滚机制

git 中存在 HEAD 的概念，HEAD 是指目前所处于的 commit。最近提交的 commit 就被称为 HEAD commit。

通过 checkout

此命令的作用是将 working directory 中文件返回到上一次 commit 时的状态。注意，此举会直接修改文件内容。

通过 reset

此命令的作用是把 staging area 中的相应文件重置到 HEAD commit，也就是最近一次 commit 的状态。主要用于 commit 误操作的状况。

reset 还可以回滚历史 commit，之前在介绍 git log 时候提到每个 commit 都有其对应的哈希码，以上命令可以回滚到历史的 commit。注意，SHA 取前 7 位，并且，执行回滚后，HEAD 的位置也将相应改变。

这条基本命令可以查看当前所有的 branch，并且了解目前所在 branch。一般来说，一个新的 branch 是 git 工程的另一个版本，它继承部分 master 的 commits 但也会包括自己独有的 commits。

以上一段命令给出了建立一个新的分支并最终将其合并入主分支的过程。

  1. 新建一个 branch 并切换过去；（branch_name 一般要能刻画分支特点）
  2. 在新建立的 branch 上增添文件并提交 commit；
  3. 将焦点切换回 master 并且把分支合并进来。注意，这里的关键在于 master 是主要的分支，合并时思考的重心应当是如何将新分支里的修改嵌入 master。同时，此时可能会出现冲突，称为 merge conflict，git 会通过不同方向的三角记号标明冲突所在位置，此时需要手动消除冲突后再次 merge。
  4. 最后合并后的分支是无用的分支，通过命令将其删除。

关于远端协作

引入 remote 概念，remote 是一个共享的 git repository，它使得多个协作者在不同地点协作同一个 git 工程。

将远端的工程克隆到本地，remote_location 为远端地址，clone_name 是自己给此工程起得名字，别人并不会知道。

remote address 会被命名为 origin，此条指令可查看远端版本。

如果协作的时间较长，可通过 fetch 将远端的修改拷贝到本地，使用 merge 指令将其合并到本地 master 中（此时也可能需要处理 merge conflict）。最后，将本地的分支 push 到远端，完成协作。
